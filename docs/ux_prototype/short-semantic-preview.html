2. Type 上显示短语义（ME-033）

无 marker — 基线

Type 无 marker 时不占用额外高度。

1 个 marker — 顶部一行

marker 居中，方括号样式，字号 11px，灰色 #515154。位于矩形顶部外侧 ~11px。

多个 marker — 一行空格分隔

多个 marker 用空格连写成一行；超宽时延长文本宽度（不换行，Fowler 书里也是一行）。

带参数 marker

带参数的写法按 schema 渲染：[key: <type>]、[n1, n2]。

3. Relation 上显示短语义（ME-034）

关键区分：一个 Relation = 两个方向相反的 映射。按 PRD：
  • 附于映射（e.g. list、class、key、historic、abstract、immutable）→ 视觉上 靠近该方向的基数标识，有 source / target 两个槽位。
  • 附于（递归）关联（e.g. hierarchy、dag、multiple_hierarchies）→ 放在 关系线中点，整条线只有一个。
  • 对"附于递归关联/映射"的标记（如 dag），统一按"附于关联"处理，放中点。
  • hierarchy 在 PRD 中有双重用法（递归关联=整条；多值映射=一端）。Phase 1 先按"整条关联"处理，避免两个语义槽位冲突；后续视反馈再加。

附于映射 — 靠近基数（推荐）

每端独立：基数在上，映射 marker 在下。两侧 marker 互不影响。

附于关联 — 整条线中点

递归关联的 [hierarchy] 放中点；基数仍贴两端。

混合：两端映射 marker + 中点关联 marker

一条线最多能同时有 3 个独立槽位：source-end / target-end / association-center。
⚠️ 决策点 2 需确认：
  1. 两端映射 marker 的排版：贴在基数下方（如示例）还是外侧？我建议下方，节省横向空间。
  2. 关联 marker 位置：线段中点 + 线的法向偏 ~10px（同 prev prototype）。
  3. 是否允许用户拖动调整 marker 位置？我建议 不做，始终自动排版。

⚠️ 决策点 3 需确认：
  1. "仅多值端"/"仅递归关联" 这种语义约束是强制的（picker 只在多值端才出现 [list]）还是只给 tooltip 提示？我建议强制，更友好。"多值"判定：source/target 基数属于 one_or_more / zero_or_more / two_or_more / range(max=null 或 max>1)。
  2. "递归关联"判定：source.typeId === target.typeId。
  3. 同一槽位是否允许同 marker 重复？我建议 不允许。key 重选覆盖 keyType。
  4. schema 里的 range（= [n1, n2]）和 PRD 一样从 UI 下拉里移除，保留在 schema 里不生效（向后兼容）？还是连 schema 一起清理？建议先只从 UI 移除，schema 清理另开一条小 story。

6. 数据模型调整

// TypeElement — 已就位
TypeElement.semantics?: ShortSemantic[]      // 可选

// RelationElement — 当前只有一个 semantics[]；要按三段拆分
RelationElement {
  source: { typeId, cardinality, cardinalityRange?, semantics?: ShortSemantic[] }  // 映射级
  target: { typeId, cardinality, cardinalityRange?, semantics?: ShortSemantic[] }  // 映射级
  associationSemantics?: ShortSemantic[]  // 关联级（中点展示）
}
需要：
· RelationEnd.semantics（新字段，映射级 marker）
· RelationElement.associationSemantics（新字段，关联级 marker）
· 废弃当前的 RelationElement.semantics（或保留作兼容回落，迁移到 associationSemantics）
· schema 同步更新；ShortSemantic.range 从 UI picker 移除
