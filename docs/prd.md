# 建模图编辑器 - 产品需求文档 (PRD)

## 1. 概述
基于Web的协作式建模图编辑器，使用Martin Fowler在《分析模式》一书中采用的符号体系（该符号体系源自James Martin和James Odell的《面向对象方法：基础》）。该工具支持实时多作者协作、多种格式导出，并可部署在Linux机器或Docker容器中。

## 2. 目标用户
- 软件架构师和设计师
- 业务分析师
- 系统建模师
- 需要协作进行系统设计的团队
- 面向对象分析的教育者和学生

## 3. 核心需求

### 3.1 图表编辑
- **符号支持**：全面支持Martin Fowler《分析模式》中的所有符号元素：
  - 类/对象表示
  - 关系类型（关联、聚合、组合、继承、依赖）
  - 多重性指示器
  - 角色名称
  - 约束和注释
  - 模式模板和构造型
- **绘图工具**：
  - 拖放形状创建
  - 关系连接工具
  - 文本标注工具
  - 对齐和分布辅助工具
  - 网格和吸附到网格
  - 缩放和平移
- **图表管理**：
  - 项目中支持多个图表
  - 图层管理
  - 撤销/重做历史记录
  - 图表元数据导入/导出

### 3.2 导出能力
- **元数据导出**：JSON/XML格式，保留图表结构、元素和关系
- **图像导出**：SVG（矢量）、PNG、PDF
- **代码生成**：可选导出为编程语言骨架（未来增强）
- **打印支持**：高质量打印，支持页面布局选项

### 3.3 实时协作
- **类Google Docs体验**：
  - 多用户同时编辑
  - 实时光标位置和选择指示器
  - 变更跟踪和版本历史
  - 评论和讨论线程
  - 用户权限（查看、编辑、管理）
- **协作功能**：
  - 通过电子邮件/链接邀请用户
  - 使用操作转换（OT）或无冲突复制数据类型（CRDT）实现实时更新
  - 离线支持，重新连接时同步
  - 在线状态指示器（谁在线、正在编辑什么）

### 3.4 项目管理
- **工作区组织**：
  - 项目文件夹和子文件夹
  - 搜索和筛选
  - 标签系统
  - 收藏/星标图表
- **共享和访问控制**：
  - 公开/私有项目
  - 团队工作区
  - 带过期时间的链接共享
  - 访问审计日志

## 4. 非功能性需求

### 4.1 性能
- 对于最多500个元素的图表，在2秒内加载完成
- 实时协作更新延迟在200毫秒内
- 典型图表导出为PNG/SVG在5秒内完成

### 4.2 可扩展性
- 每个图表至少支持50个并发用户
- 协作服务器支持水平扩展
- 针对大型图表集的数据库优化

### 4.3 可靠性
- 核心服务可用性99.9%
- 数据持久化，自动备份
- 并发编辑冲突解决

### 4.4 安全性
- 端到端加密敏感数据
- OAuth 2.0 / SSO集成
- 基于角色的访问控制（RBAC）
- 合规性审计跟踪

### 4.5 可用性
- 直观的用户界面，学习曲线最小
- 为高级用户提供键盘快捷键
- 响应式设计，支持平板/桌面
- 可访问性符合WCAG 2.1 AA标准

## 5. 技术架构

### 5.1 高层架构
```
┌─────────────────┐    ┌─────────────────┐    ┌─────────────────┐
│    Web客户端    │◄──►│   API网关       │◄──►│   协作服务      │
│   (React/Vue)   │    │  (负载均衡器)   │    │                 │
└─────────────────┘    └─────────────────┘    └─────────────────┘
         │                       │                       │
         │                       │                       │
┌─────────────────┐    ┌─────────────────┐    ┌─────────────────┐
│   静态资源      │    │   应用服务       │    │   实时消息      │
│   (CDN)         │    │                 │    │                 │
└─────────────────┘    └─────────────────┘    └─────────────────┘
                               │
                               │
                     ┌─────────────────┐
                     │   数据层        │
                     │   (数据库, 缓存) │
                     └─────────────────┘
```

### 5.2 组件分解
- **前端**：现代JavaScript框架（React/Vue/Angular）配合图表库（可能自定义或基于mxGraph、JointJS等）
- **后端服务**：
  - 用于CRUD操作的REST/GraphQL API
  - 实时协作服务（WebSocket）
  - 认证和授权服务
  - 导出服务（图像生成）
- **数据存储**：
  - 主数据库（PostgreSQL/MySQL）用于用户数据和元数据
  - 文档存储（MongoDB）用于图表内容
  - Redis用于缓存和实时状态
- **基础设施**：
  - Docker容器化
  - Kubernetes编排（可选）
  - Nginx反向代理
  - CI/CD流水线

### 5.3 技术栈推荐
- **前端**：React + TypeScript + Redux/MobX
- **图表库**：使用Paper.js或Fabric.js的自定义Canvas实现，或利用成熟的库如JointJS
- **实时通信**：Socket.io或WebSocket配合ShareDB/Yjs用于协作
- **后端**：Node.js/Express或Python/FastAPI
- **数据库**：PostgreSQL用于关系数据，Redis用于缓存
- **导出**：Puppeteer/Headless Chrome用于PNG/PDF，svg2png用于图像转换
- **部署**：Docker + Docker Compose，可选Kubernetes

## 6. 部署选项

### 6.1 Docker部署
- 单个`docker-compose.yml`包含所有服务
- 环境变量配置
- 数据持久化卷
- 通过Docker镜像标签轻松更新

### 6.2 Linux机器部署
- 每个组件的Systemd服务文件
- Nginx作为反向代理，支持SSL终止
- PostgreSQL/Redis作为系统包或容器
- 备份脚本和监控

### 6.3 云原生部署
- Kubernetes清单用于扩展
- 云存储用于资产
- 托管数据库（RDS, Cloud SQL）
- CDN用于静态资源

## 7. 开发里程碑

### 第一阶段：基础（第1-4周）
- 具有核心符号元素的基本图表编辑器
- 单用户编辑
- 导出为SVG/PNG
- 仅本地存储

### 第二阶段：协作（第5-8周）
- 用户认证
- 实时多用户编辑
- 基本在线状态指示器
- 项目管理

### 第三阶段：完善（第9-12周）
- 高级符号元素
- 元数据导入/导出
- 增强的协作功能
- 性能优化

### 第四阶段：部署（第13-16周）
- Docker打包
- 生产环境部署脚本
- 监控和日志记录
- 文档编写

## 8. 风险与依赖

### 技术风险
1. **实时协作复杂性**：OT/CRDT实现非易事
2. **图表渲染性能**：大型图表可能导致浏览器变慢
3. **导出质量**：高保真导出需要仔细实现

### 缓解策略
- 使用成熟的协作库（Yjs, ShareDB）
- 为大型图表实现虚拟滚动
- 早期使用复杂图表测试导出功能

### 依赖
- **开源库**：图表渲染、实时通信
- **浏览器兼容性**：支持Canvas和WebSocket的现代浏览器
- **基础设施**：Docker、Linux服务器访问权限

## 9. 成功指标
- 用户参与度：日活跃用户、每个用户创建的图表数
- 性能：页面加载时间、协作延迟
- 可靠性：正常运行时间、错误率
- 用户满意度：NPS分数、功能采用率
